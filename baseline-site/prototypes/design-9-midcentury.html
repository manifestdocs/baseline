The type is the specification.

A statically typed, effect-aware language where types encode correctness, side effects are explicit capabilities, and the compiler proves the rest.

Get Started View on GitHub 
// Types encode your intent.
// Effects constrain your dependencies.

type Port = Int where self > 0 and self <= 65535

serve! : Port -> {Console, Http} ()
serve! = |port| {
  Console.println!("Listening on ${port}")
  Router.new()
    |> Router.get("/users", list_users)
    |> Router.post("/users", create_user)
    |> Router.listen!(port)
}
Why Baseline

Correct by construction.

Effects are capabilities

A function typed {Console, Http} can print and make HTTP calls, and nothing else. Side effects are declared in type signatures and enforced by the compiler. No dependency can access the filesystem unless you grant it {Fs}. The default is no access.

Refinement types

type Port = Int where self > 0 proves constraints at compile time, not at runtime. Define what valid data looks like once. The compiler checks every use. No defensive re-validation. No scattered checks. No "someone else must have validated this" assumptions.

No null, no exceptions

Option<T> and Result<T, E> are the only error paths. Pattern matching is exhaustive -- the compiler rejects code that leaves a branch unhandled. If a function can fail, its signature says so. If it returns a value, that value exists.

One way to do each thing

One syntax for errors, one for effects, one for data transformation. Pipes over nesting. Combinators over loops. Interpolation over concatenation. Fewer choices means fewer mistakes -- and code that every team member and every LLM reads the same way.

Code

See it in action.

Pattern matching

Exhaustive by default. The compiler rejects unhandled branches. Model your domain with sum types and let the type system ensure every case is covered. No fall-through, no forgotten edge cases. 

type Connection =
  | Disconnected
  | Connected(Socket)
  | Error(String)

handle : Connection -> String
handle = |conn| match conn
  Disconnected      -> "reconnecting"
  Connected(socket) -> "active"
  Error(msg)       -> "failed: ${msg}"

Pipes and transforms

The pipe operator |> chains transformations left to right. Data flow reads like prose. Combinators replace loops for filtering, mapping, and folding -- keeping logic pure and composable. 

active_users : List<User> -> List<String>
active_users = |users|
  users
  |> List.filter(|u| u.active == true)
  |> List.map(|u| u.name)

total_age : List<User> -> Int
total_age = |users|
  users
  |> List.map(|u| u.age)
  |> List.fold(0, |acc, x| acc + x)

Tracked effects

Every side effect is declared in the type signature. {Http, Console} means this function can fetch data and print, and nothing else. The ? operator propagates errors. Pure functions need no effect annotation at all. 

fetch_user! : Int -> {Http, Console} Result
fetch_user! = |id| {
  Console.println!("Fetching user ${id}")
  let response = Http.get!("/users/${id}")?
  match response.status
    200 -> Ok(response.body)
    404 -> Err("User not found")
    _   -> Err("Unexpected: ${response.status}")
}

Inline tests

Tests live next to the code they verify, using where blocks. The compiler runs them with blc test. Functions and their tests are always in sync because they share the same file and scope. 

add : (Int, Int) -> Int
add = |a, b| a + b
  where
    test "adds positive numbers" = add(1, 2) == 3
    test "handles zero"          = add(0, 5) == 5
    test "handles negatives"     = add(-1, 1) == 0

safe_divide : (Int, Int) -> Result
safe_divide = |a, b|
  if b == 0 then Err("Division by zero")
  else Ok(a / b)
  where
    test "divides"  = safe_divide(10, 2) == Ok(5)
    test "by zero"  = safe_divide(1, 0) == Err("Division by zero")
Features

Language at a glance.

Algebraic Effects
Side effects declared in type signatures and enforced by the compiler. Built-in: Console, Http, Fs, Log, Time.
Refinement Types
Int where self > 0 proves constraints at compile time. Parse, don't validate.
Pattern Matching
Exhaustive matches on sum types, records, lists, and tuples. Guards, or-patterns, destructuring.
Pipe Operator
|> chains data transformations left to right. The compiler warns against deeply nested calls.
Sum Types
Model domain variants precisely. Option<T> and Result<T, E> are built-in.
Records
Named fields, dot access, update syntax, structural typing with row polymorphism.
Inline Tests
where test "name" = expr keeps tests alongside the code they verify.
ML-Style Functions
name : Sig = body. Lambdas with |x| expr. Type inference for locals.
Modules
@module Name with explicit imports. Prelude levels from none to server.
Error Propagation
The ? operator propagates errors. match for explicit handling. No try/catch.
JSON Diagnostics
All compiler output is structured JSON. LLMs self-correct against machine-readable errors.
LLM-Native Design
Unambiguous syntax, constrained solution space, structured errors. Designed for machine generation.
Positioning

What changes.

Baseline synthesizes proven ideas from ML, Rust, Koka, Liquid Haskell, and Elm. The combination is new. The ingredients are battle-tested.

Concern Status Quo Baseline
Error handling Exceptions, null, unchecked tuples Result<T, E> and Option<T> only. Exhaustive matching.
Side effects Implicit, unconstrained, invisible Declared in type signatures. Compiler-enforced.
Validation Scattered runtime checks, defensive code Refinement types prove constraints at compile time.
AI code gen Best-effort, validate with tests Types constrain correctness. Effects sandbox access.
Code style Many ways, style guides, linter wars One way to do each thing. Compiler-enforced.
Testing Separate files, separate frameworks Inline tests with where blocks. Always in sync.
Get Started

Up and running.

1

Install

brew install baseline-lang/tap/blc
2

Write

Create hello.bl:

@prelude(script)

greet : String -> String
greet = |name| "Hello, ${name}!"

main! : () -> {Console} ()
main! = {
  let message = greet("World")
  Console.print!(message)
}
3

Run

blc run hello.bl
# Hello, World!

blc check hello.bl --json
# {"status": "ok", "diagnostics": []}
Influences

Standing on shoulders.

Baseline draws from decades of programming language research. Each influence contributes a specific, well-understood idea.

  • ML / OCaml -- type inference, pattern matching, algebraic data types
  • Rust -- error handling, tooling quality
  • Koka -- algebraic effects
  • Liquid Haskell -- refinement types
  • Elm -- friendly errors, simplicity
  • F# -- pragmatic functional programming

Start building.

The compiler, grammar, and core type system are functional. Baseline is in active development and accepting contributors.
