The Language in 60 Seconds

Safe. Correct. Connected.

The type is the specification

Constraints live in types, not in runtime checks scattered across a codebase. Port is not just an Int — it is an Int proven to be between 1 and 65535. Any function that accepts a Port receives a value the compiler has already verified.

Parse and validate at the boundary; trust the interior. External input enters through parsing functions that produce refined types. Internal code operates on values whose constraints are already proven. 

// The type IS the specification
type Port = Int where 1 <= self <= 65535
type Percentage = Int where 0 <= self <= 100

// Any function accepting Port gets a
// compiler-verified value. No checks needed.
configure : Port → String
configure = |port|
  "Listening on ${port}"

Effects are capabilities

Every side effect is declared in the type signature. A function typed {Console, Http} can print and make HTTP calls — and nothing else. The compiler rejects any operation outside the declared capabilities.

This makes the effect system a security boundary. When AI-generated or third-party code executes, its capabilities are visible in the type signature and enforced by the compiler. Not a convention — a guarantee. 

// Effects declared in the signature
fetch_and_log! : String → {Http, Console} ()
fetch_and_log! = |url| {
  let body = Http.get!(url)
  Console.println!(body)
}
// Fs.delete!("data.txt")
// ^ COMPILE ERROR: Fs not in effect set

Error handling without exceptions

Result<T, E> replaces exceptions. Option<T> replaces null. Pattern matching is exhaustive — the compiler rejects code that doesn't handle every case. No try/catch, no null pointer exceptions, no silent failures. 

// Every case handled. Compiler enforced.
describe : Result<String, String> → String
describe = |result|
  match result
    Ok(value) → "Got: ${value}"
    Err(msg) → "Failed: ${msg}"
  // Miss a case? Compiler error.

Built for Connected Systems

HTTP-Native by Design

Route handlers are typed functions with declared effects. HTTP is part of the language, not a framework bolted on after the fact.

Complete HTTP Server server_test.bl 
@prelude(server)

hello_handler : { method: String, path: String, body: String } → {Http} Result<String, String>
hello_handler = |req| Ok("Hello, World!")

echo_handler : { method: String, path: String, body: String } → {Http} Result<String, String>
echo_handler = |req| Ok(req.body)

main! : () → {Http, Console} ()
main! = || {
  let app = Router.new()
    |> Router.get("/hello", hello_handler)
    |> Router.post("/echo", echo_handler)
  Server.listen!(app, 3000)
}
A complete HTTP server. Handlers are typed functions with declared effects. Routes compose with the pipe operator. This is real, working Baseline code.
Pipe Composition router.bl 
build_app : () → { routes: List<Unknown> }

    |> Router.post("/echo", echo_handler)
}

Designed for the Agentic Age

When Machines Write the Code, the Language Matters More

Most programming languages were designed for humans writing code by hand. When an LLM generates code, it must navigate ambiguity, implicit behavior, and multiple valid ways to express the same operation. Baseline constrains the solution space by design. One way to handle errors (Result). One way to express side effects (the effect set). One way to structure data (records and sum types). The type signature IS the spec — if the LLM gets the type right, the implementation is constrained to be correct.

The effect system is a security boundary. When an AI agent generates Baseline code, you know exactly what it can do by reading the type signature. {Console, Http} means it can print and make HTTP calls — not delete files, not access the filesystem. This is not a convention. It is compiler-enforced.

Structured JSON output from the compiler enables automated self-correction loops: the LLM generates code, the compiler checks it, structured errors flow back, the LLM fixes the issues, and the cycle repeats until the types check. No test suite required for basic correctness — the type system does the work.

The Compiler Experience

Catch. Explain. Fix.

Effect Not Declared blc check

  process! : String → {Log} String
  process! = |data| Http.get!(data)
                     ^^^^^^^^

  Http.get! requires the {Http} effect,
  but the signature only declares {Log}.

  Hint: Add Http to the effect set:
    process! : String → {Log, Http} String
Inspired by Elm's error messages. Every diagnostic tells you what went wrong, where, and how to fix it. Structured JSON output (blc check --json) enables automated tooling and LLM self-correction loops.

Everything You Need. Nothing You Don't.

Baseline ships the features that matter for building reliable connected systems.

Refinement Types

Encode value constraints in types. Port = Int where 1 <= self <= 65535. Checked at compile time.

Algebraic Effects

Track every side effect in type signatures. The compiler enforces what is permitted.

Pattern Matching

Exhaustive by default. Every branch handled. No fall-through, no missed cases.

Pipe Operator

|> chains transformations left to right. Clear data flow, no nesting.

Sum Types

Model domain variants precisely. Result, Option, and your own discriminated unions.

Records

Structural typing with named fields. Dot access. Type-safe construction.

Built-in HTTP

Server, router, and client built into the language. Type-checked handlers with declared effects.

String Interpolation

"Hello ${name}" with type-safe expressions. Clean, readable output formatting.

Prelude System

Six levels from None to Server. Progressive capability loading — use only what you need.

Friendly Errors

Elm-inspired diagnostics. What went wrong, where, and how to fix it.

Effect Sandbox

Effect system as capability boundary. Know exactly what any function can do.

JSON Diagnostics

blc check --json for structured compiler output. Enables automated LLM self-correction loops.
